## 為什麼
- 目前的記憶體啟發式（例如 `LOW_MEMORY_THRESHOLD = 25KB`）是為 ESP32-C3 固定寫死，對於記憶體更大的板子（ESP32-S3 擁有 512KB 以上可用堆積）會過度保守，而在功能擴充時又過於樂觀。
- 緩衝池數量與串流分塊尺寸在編譯期固定，造成輕量化建置（模擬、輕量除錯）無法釋放 RAM，重度建置（S3 生產環境）也無法善用更大的緩衝。
- 運維與遙測端點無法得知目前啟用的記憶體策略，使跨環境追查記憶體退化變得困難。

## 調整內容
- 建立 `MemoryProfile` 設定模組，依硬體特徵（如 `ESP32C3_SUPER_MINI`、PSRAM 狀態、建置旗標）推導各項閾值、緩衝池數量與串流分塊大小。
- 重構 `MemoryManager`、`BufferPool`、`StreamingResponseBuilder`，改為讀取現行配置檔而非硬編碼常數，並支援 OTA 切換韌體型態時的執行期重新載入。
- 透過 `/api/memory/detailed` 及監控儀表板揭露目前的記憶體配置檔與有效限制，且在啟動時記錄所選配置檔。
- 更新相關文件（docs 與 README），讓貢獻者了解如何調整或新增配置檔，並增加保護性測試／診斷工具，確認各個 PlatformIO 環境載入正確配置。

## 影響
- 在低記憶體板子上收斂緩衝池仍能維持除錯環境的反應速度，同時允許 ESP32-S3 生產版使用更大的緩衝以改善 UI 體驗。
- 讓記憶體調校與硬體配置一一對應，降低回歸風險並簡化後續最佳化。
- 對通訊協定／控制邏輯的風險極低；主要變動限定於記憶體優化子系統與遙測層，既有 API 維持相容。
